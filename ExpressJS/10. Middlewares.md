# Middleware in Express

A function that runs *before* your main route handler. That's it.

Used to pull out repeated logic — auth, validation, logging, rate limiting, parsing — so you're not rewriting it in every route.

---

## Syntax

```js
const logStuff = (request, response, next) => {
  console.log(...)
  next()
}
```

Three params, auto-passed by Express:

- `request` / `response` — same as any route handler, full access to all methods and properties
- `next` — a function that passes control to whatever comes next in the chain (another middleware, or the final route handler). Call it when your middleware is done. Don't call it, nothing after runs.

> Middleware is just a request handler function with `next` added. That's the only real difference.

---

## Two Ways to Use It

### 1. Global — `app.use()`

```js
app.use(logStuff)

app.get('/users', ...) // logStuff runs here
app.post('/users', ...) // and here
```

Every route defined *after* the `app.use()` call will run that middleware. Order matters.

---

### 2. Route-specific

Pass it as an argument between the route string and the handler:

```js
app.get('/users/:id', resolveUserIdx, (req, res) => {
  // resolveUserIdx ran first
})
```

Chain multiple:

```js
app.get('/users/:id', middleware1, middleware2, (req, res) => {
  // runs in order: middleware1 → middleware2 → handler
})
```

Or inline anonymous:

```js
app.get('/users/:id', (req, res, next) => {
  // inline middleware
  next()
}, (req, res) => {
  // main handler
})
```

---

## Passing Data Between Middleware

`request` is the same object all the way down the chain. So if you add a property in middleware, the next function can read it.

```js
request.userIdx = userIdx // set it in middleware
// ...
next()

// in the route handler:
const idx = request.userIdx // already there
```

---

## Real Example

```js
// resolves a user index from :id param, handles errors early
const resolveUserIdxById = async (request, response, next) => {
  const parsedId = parseInt(request.params.id)

  if (isNaN(parsedId)) return response.sendStatus(400) // bad request, stop here

  const users = await readDB()
  const userIdx = users.findIndex((el) => el.id === parsedId)

  if (userIdx === -1) return response.sendStatus(404) // not found, stop here

  request.userIdx = userIdx // attach to request for the next handler
  next()
}
```

```js
app.get('/users/:id', resolveUserIdxById, (req, res) => {
  // no id parsing, no error checking — resolveUserIdxById already handled it
  // req.userIdx is ready to use
})
```

This is the main reason middleware exists — keep route handlers clean, handle the boring/repeated stuff once.

---

## `next(error)` — Error Handling

Passing anything to `next()` signals an error to Express. It skips all remaining middleware and jumps straight to your error handler:

```js
next(new Error('Something went wrong'))
```

Your error handler is just a middleware with a 4th param `err` as the first argument. That signature is how Express knows it's an error handler:

```js
app.use((err, req, res, next) => {
  res.status(500).json({
    message: err.message,
  })
})
```

> Register your error handler at the very bottom of your app, after all routes. Express only reaches it via `next(err)`.
